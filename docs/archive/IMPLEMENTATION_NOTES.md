# AgentGuard — Implementation Notes

> **Document version:** 1.0  
> **Date:** March 2026  
> **Classification:** Confidential — AgentGuard Internal  
> **Purpose:** Explain why the implementation diverged from the original architecture documents, and what the migration path looks like at scale.

---

## Why This Document Exists

The original `ARCHITECTURE.md` (v2.0) and `ROADMAP.md` (v2.0) described a target architecture with Hono, Prisma, AWS ECS, Redis, BullMQ, and Next.js. What was actually built used Express, raw SQL, Azure Container Apps, no cache, no queue, and a static HTML dashboard.

This document records the rationale for each divergence. It is honest about the tradeoffs. It does not pretend the original docs were intentionally aspirational — they were written before the implementation was complete and reflected design intent that the build didn't follow.

The architecture that was built is coherent, maintainable, and has shipped a working product. The decisions below were pragmatic, not accidental.

## Why Express Instead of Hono

**What the architecture said:** Hono 4.x — faster benchmarks, edge-compatible, TypeScript-first middleware.

**What was built:** Express 5.x.

**Reason:** The initial API implementation was generated by an AI sub-agent. Sub-agents trained on general Node.js patterns default to Express — it is the most widely understood framework, with the deepest documentation coverage. Hono was a deliberate architecture-doc choice, but the sub-agent didn't follow it, and the Express implementation was functional and tested before the discrepancy was noticed.

**Why we kept it:** Rewriting a working server to a different framework is a risk-heavy week for zero user-visible benefit at current scale (< 1,000 req/s). Express 5.x is production-mature, has first-class TypeScript support via `@types/express`, and runs on Node.js 22 without compatibility issues.

**Trade-offs accepted:**
- Hono is faster on microbenchmarks (~2–3x on simple routes). At current traffic volumes this is not measurable.
- Hono is edge-compatible (Cloudflare Workers, Deno Deploy). Not relevant for Azure Container Apps.
- Express error handling is less elegant than Hono's `c.json()` chaining. Acceptable with good route discipline.

**Migration path:** Express → Hono migration is a drop-in refactor — same routes, same middleware, different handler signatures. If scaling to 10,000 concurrent tenants makes Express overhead visible in profiling, the migration is straightforward and the API contract stays identical.

## Why Raw SQL Instead of Prisma

**What the architecture said:** Prisma 5.x ORM with `@prisma/adapter-pg`. Schema defined in `prisma/schema.prisma`; TypeScript types derived via `prisma generate`.

**What was built:** Raw parameterised SQL via the `IDatabase` typed interface.

**Reason:** The schema is deliberately simple — 12 tables, mostly CRUD, no complex relationships. Prisma's value proposition is strongest when you have 30+ models with complex associations and frequent migrations. For this schema, Prisma adds:
- A build step (`prisma generate`)
- Prisma-specific query syntax that differs from SQL
- A `schema.prisma` file that must stay in sync with the actual database
- Operational complexity (Prisma Migrate vs. raw SQL migrations)

The `IDatabase` typed interface achieves the same TypeScript safety goal — all return types are explicitly typed, all parameters are typed — without the ORM layer.

**Trade-offs accepted:**
- No automatic migration runner. Schema changes require manual SQL DDL. At 12 tables, this is manageable.
- Prisma's query builder is more ergonomic for complex filtering and pagination. Compensated by explicit SQL (which is easier to understand, optimise, and audit).
- No `prisma studio` for local DB browsing. Compensated by standard `psql` or any PostgreSQL GUI.

**Migration path at 10K+ tenants:**

If the schema grows to 30+ models, or if a new engineer joins who is more productive with Prisma, migration is straightforward:

1. Generate a `prisma/schema.prisma` from the existing DDL (`prisma db pull`)
2. Run `prisma generate` to get the typed client
3. Replace `IDatabase` method implementations with Prisma calls (one adapter file)
4. The route handlers don't change — they call `IDatabase` methods, not ORM-specific syntax

Alternatively, Drizzle ORM is a closer fit to the raw-SQL approach (schema-as-code, typed results, no code generation required at runtime).

## Why Azure Instead of AWS

**What the architecture said:** AWS ECS Fargate, ECR, RDS PostgreSQL with PgBouncer, ElastiCache Redis Sentinel.

**What was built:** Azure Container Apps, Azure Container Registry, Azure Database for PostgreSQL.

**Reason:** Founder preference. The founder has existing Azure credits, familiarity with Azure tooling, and a preference for Azure's Australian regional presence (lower latency for AU-based early users). The product is architecturally cloud-agnostic — it runs Docker + PostgreSQL, both of which are available on every major cloud and on-premises.

**Why Azure Container Apps specifically:** Container Apps (not Azure Kubernetes Service) was chosen for the same reason the architecture doc chose ECS over EKS — simpler operations for a small team. Container Apps handles scaling, ingress, and SSL without requiring cluster management.

**Trade-offs accepted:**
- AWS has a larger marketplace and more mature AI/ML tooling (Bedrock, SageMaker). Relevant for future marketplace listings.
- Some enterprise customers prefer AWS deployments. The Terraform/GitHub Actions pipeline can be adapted to AWS in a week.
- ElastiCache Redis and RDS are not available. Azure Cache for Redis and Azure Database for PostgreSQL are drop-in functional equivalents.

**Migration path:** The application is container-based and database-agnostic. A full AWS migration (ECR, ECS Fargate, RDS) would take 1–2 weeks of DevOps work, predominantly:
1. Rewriting Terraform from Azure to AWS (or writing new AWS CDK/CloudFormation)
2. Updating GitHub Actions workflow (OIDC target, push command)
3. Reprovisioning the database (export → import)
4. Updating DNS at Cloudflare

No application code changes required.

## Why No Redis (Yet)

**What the architecture said:** ElastiCache Redis 7 Sentinel mode for: kill switch flags, HITL gate state, rate limit counters, policy bundle cache (TTL 60s). BullMQ for background jobs.

**What was built:** No Redis. No background queue.

**Reason:** Adding Redis adds a managed infrastructure component, operational monitoring, connection pool management, and failure modes. At current scale (< 100 concurrent tenants), the cost-benefit is negative:

- **Rate limits:** Implemented as an in-memory sliding window (satisfactory for single-container deployments)
- **Webhook delivery:** Fire-and-forget synchronous HTTP call from the request handler (< 200ms overhead; acceptable for current traffic)
- **Kill switch:** Stored in PostgreSQL; propagates within the next evaluate call (acceptable latency for current use cases)
- **Policy bundles:** Loaded at startup; no per-request cache needed at current scale

**Trade-offs accepted:**
- In-memory rate limits are not shared across container instances. If Container Apps scales to 2+ replicas, rate limits become per-instance. This is a real limitation.
- Webhook delivery has no retry mechanism. If the target endpoint is down, the event is lost.
- Kill switch propagation is not real-time — it takes effect on the next `/evaluate` call, not within seconds.

**Migration path at scale:**

Adding Redis (Azure Cache for Redis) would change:

1. Rate limit counters: `INCR` + `EXPIRE` in Redis, shared across all instances
2. Webhook delivery: push to BullMQ queue (persisted in Redis); background worker process handles retries
3. Kill switch: Redis key `killswitch:{tenantId}:{agentId}` with 24h TTL; SDK polls `GET /sdk/kill-switch` via long-poll backed by Redis subscription
4. Policy bundle cache: Redis key `policy-bundle:{tenantId}:{policyId}` with 60s TTL

The `IDatabase` interface is not involved in Redis operations — Redis is a separate client. Route handlers would need minimal changes (swap in-memory counter for Redis INCR call).

## Why a Static Dashboard Instead of Next.js

**What the architecture said:** Next.js 14 App Router + shadcn/ui + Tailwind CSS + SWR + Recharts + WebSocket real-time updates.

**What was built:** Static HTML + Vanilla JavaScript.

**Reason:** The current user population for the dashboard is: founders, internal team, design partners (< 20 people). A static HTML dashboard with inline JavaScript delivers the necessary functionality — agent list, audit log, policy editor, kill switch — without a build pipeline, Node.js server, or framework upgrade path to manage.

Next.js adds:
- A build step (`next build`)
- A separate Node.js server process
- React component complexity for what is essentially server-rendered content
- A `package.json` with 20+ additional dependencies
- Framework upgrade maintenance

**Trade-offs accepted:**
- No React component library. UI is hand-written HTML — less polished.
- No SWR data fetching — polling implemented manually.
- No WebSocket real-time updates — dashboard refreshes on user action.
- Harder to add complex interactive features (e.g., real-time risk charts) without a component framework.

**Migration path:**

When the product has external enterprise users who require a polished, real-time dashboard, the migration to Next.js is a frontend project that doesn't touch the API:

1. Create `apps/dashboard/` with Next.js App Router
2. Re-implement existing pages as React Server Components + Client Components
3. Add WebSocket connection to `GET /api/v1/events/stream`
4. Use shadcn/ui for design system
5. Deploy to Azure Static Web Apps or Vercel (both support Next.js)

No backend changes required. The API contract is stable.

## Migration Path: 10K+ Tenants

If AgentGuard reaches 10,000 active tenants (approximately $5M ARR at current pricing), the following changes would be needed. None require rewriting the core product — they are infrastructure and operational scaling:

### Database

**Current:** Single PostgreSQL instance (4 vCPU, 16GB on Azure), direct `pg` pool (max 20 connections).

**At 10K tenants:**
1. Add PgBouncer transaction-mode connection pooler (reduces PostgreSQL connections from 20 per container to pooled total)
2. Add read replicas for audit log queries (heavy read workload)
3. Consider ClickHouse or TimescaleDB for `audit_events` (time-series optimised, dramatically faster for large-window queries)
4. Add Row-Level Security (PostgreSQL RLS) as defence-in-depth

**Schema changes:** Minimal. Add `organisation_id` for multi-org tenants. Partition `audit_events` by `tenant_id` + month.

### Caching

**Current:** None.

**At 10K tenants:**
1. Policy bundle cache in Redis (60s TTL) — eliminates repeated policy YAML parsing
2. Rate limit counters in Redis — shared across container replicas
3. Session/kill switch state in Redis — sub-100ms propagation

### Background Workers

**Current:** Inline webhook delivery.

**At 10K tenants:**
1. BullMQ (or Azure Service Bus) queue for webhook delivery with exponential backoff retry
2. Async SIEM push workers (Splunk HEC, Sentinel DCR)
3. Cost aggregation background jobs (hourly rollup)

### Application

**At 10K tenants:**
1. Express → Hono migration (if profiling shows framework overhead at scale)
2. Full Zod validation on all endpoints (already in progress)
3. JWT / RS256 for human user auth (scales better than per-request key lookup)
4. API key bcrypt hashing (required before 10K — a security prerequisite, not a scale issue)

### Infrastructure

**At 10K tenants:**
1. Multi-region deployment (Azure AU East + at least one EU/US region for data residency)
2. Kubernetes (AKS or EKS) if Container Apps scaling limits are hit
3. WAF (Cloudflare Pro tier or Azure WAF) for DDoS and OWASP protection
4. Centralised log aggregation (Datadog or Azure Monitor)
5. Automated canary deployments (vs. current direct push to production)
