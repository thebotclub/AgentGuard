### Core R&D Activity 1: Sub-Millisecond Policy Evaluation Engine

**Activity description:**

✅ **Pre-filled:**

> **Objective:** Design and develop a runtime policy evaluation engine capable of intercepting AI agent tool calls and evaluating them against configurable security policies in sub-millisecond time, without introducing meaningful latency to agent workflows.
>
> **Technical uncertainty:** At the commencement of this activity, it was not known whether a policy evaluation system could:
> (a) Achieve sub-millisecond evaluation times while supporting complex, composable policy rules (rate limiting, argument validation, pattern matching, permission scoping) across diverse tool-call schemas;
> (b) Maintain consistent performance characteristics when processing high-throughput concurrent agent tool calls without degradation;
> (c) Support real-time policy hot-reloading without requiring agent restart or causing evaluation gaps during policy transitions.
>
> **Knowledge gap:** Existing approaches to API gateway security (e.g., rate limiters, WAFs) operate on HTTP request/response patterns and are not designed for the semantic richness of AI agent tool calls, where the "request" is a natural-language-derived function invocation with arbitrary argument structures. No established methodology existed for sub-millisecond policy evaluation at this semantic layer.
>
> **Systematic progression of work:** The company conducted a series of experiments including:
> - Benchmarking multiple policy evaluation architectures (rule-based, tree-based, compiled policy graphs) against latency and throughput targets
> - Iterative prototyping of policy compilation strategies to pre-compute evaluation paths
> - Performance profiling under simulated concurrent agent workloads to identify bottlenecks
> - A/B comparison of synchronous vs. asynchronous evaluation pipelines
>
> **Outcome:** The experiments resulted in a policy engine achieving sub-millisecond evaluation, validated through systematic benchmarking. The specific architecture and optimisation techniques were not determinable in advance.

### Core R&D Activity 2: Hash-Chained Cryptographic Audit Trail

✅ **Pre-filled:**

> **Objective:** Develop a tamper-evident audit logging system for AI agent tool calls that uses cryptographic hash chaining to ensure the integrity and immutability of audit records, suitable for regulatory compliance (e.g., EU AI Act Article 12 logging requirements).
>
> **Technical uncertainty:** At the commencement of this activity, it was not known whether:
> (a) A hash-chained audit trail could be maintained in real-time alongside sub-millisecond policy evaluation without introducing unacceptable latency or creating a bottleneck in the agent execution pipeline;
> (b) The hash chain integrity could be preserved across agent restarts, crashes, and concurrent multi-agent environments without gaps or corruption;
> (c) The audit format could capture sufficient semantic context (tool name, arguments, policy decision, reasoning) while remaining efficiently queryable for compliance reporting.
>
> **Knowledge gap:** While hash-chaining is established in blockchain contexts, its application to real-time AI agent audit trails — where events occur at high frequency, in concurrent execution environments, and must capture semantic agent-specific context — had not been demonstrated. The intersection of cryptographic integrity guarantees with real-time agent monitoring performance constraints represented a novel technical challenge.
>
> **Systematic progression of work:**
> - Evaluated multiple hash-chaining strategies (linear chains, Merkle trees, hybrid approaches) for suitability in high-frequency event logging
> - Tested chain recovery mechanisms for crash resilience and restart scenarios
> - Prototyped concurrent chain management for multi-agent environments
> - Benchmarked audit write overhead against policy evaluation latency budgets
> - Iterated on audit record schema to balance semantic richness with storage efficiency

### Supporting R&D Activity 1: Multi-Framework Integration Layer

✅ **Pre-filled:**

> **Objective:** Develop a generalised integration layer enabling AgentGuard's policy engine and audit trail to intercept tool calls across 7 heterogeneous AI agent frameworks (LangChain, CrewAI, AutoGen, OpenAI Agents SDK, Vercel AI SDK, Google A2A, Express/Fastify), each with different architectures, lifecycle models, and tool-call conventions.
>
> **Relationship to core activities:** This activity directly supports Core Activities 1 and 2 by enabling the policy engine and audit trail to operate across the diverse agent framework ecosystem. Without this integration layer, the core R&D outputs would be limited to a single framework.
>
> **Technical work involved:**
> - Reverse-engineering undocumented or poorly documented hook points in each framework's tool execution pipeline
> - Designing a framework-agnostic abstraction layer that normalises tool-call representations across fundamentally different architectures (callback-based, middleware-based, event-driven, decorator-based)
> - Resolving framework-specific concurrency models (async/await, threading, event loops) to ensure consistent policy evaluation and audit logging behaviour
> - Iterative testing across framework version updates to maintain compatibility

### Supporting R&D Activity 2: Cross-Language SDK Development (TypeScript + Python)

✅ **Pre-filled:**

> **Objective:** Develop functionally equivalent SDKs in TypeScript and Python that expose identical policy configuration, evaluation, and audit interfaces while accommodating fundamental language-level differences in type systems, concurrency models, and ecosystem conventions.
>
> **Relationship to core activities:** This activity supports Core Activities 1 and 2 by making the policy engine and audit trail accessible to the two primary languages used in AI agent development. Without dual-SDK support, the core R&D outputs would reach only a subset of the target developer population.
>
> **Technical work involved:**
> - Designing a language-agnostic policy specification format that compiles to optimal evaluation code in both TypeScript and Python
> - Resolving differences in async execution models (Node.js event loop vs. Python asyncio) while maintaining consistent sub-millisecond performance guarantees
> - Ensuring hash-chain audit trail integrity across both language implementations with identical cryptographic behaviour
