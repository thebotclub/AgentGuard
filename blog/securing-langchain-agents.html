How to Secure Your LangChain Agents:
A Practical Guide

LangChain makes it easy to give your agent tools — web search, code execution, database access, API calls. That's exactly the problem. Every tool is an attack surface. This guide walks you through 5 concrete steps to add runtime security, real-time monitoring, and a kill switch to any LangChain agent.

Why LangChain Agents Need Runtime Security

LangChain's tool abstraction is genuinely elegant. You give the agent a list of tools, a goal, and it figures out how to use them. That design philosophy — maximum agent autonomy — is what makes LangChain powerful. It's also what makes unsecured LangChain agents a production risk.

When you give an agent a bash_tool, a write_file tool, or a Stripe API wrapper, you're not just giving it capabilities. You're expanding the attack surface for anyone who can influence that agent's inputs. And in a LangChain agent, inputs come from everywhere: user messages, web search results, database queries, retrieved documents, outputs from other agents.

⚠ The Fundamental Problem

LangChain's system prompt is probabilistic, not deterministic. You can write "Never access /etc/passwd" in your system prompt and an adversarial user or poisoned tool output can still cause the agent to attempt it. System prompts are guidance to an LLM. Runtime policies are enforced controls. They are not the same thing.

Here's a minimal example of a LangChain agent with tool access — and what an unsecured version looks like:

Python · Vulnerable Pattern
# Don't do this in production
from langchain.agents import AgentExecutor, create_openai_functions_agent
from langchain_openai import ChatOpenAI
from langchain.tools import BashTool, WriteFileTool, WebSearchTool

llm = ChatOpenAI(model="gpt-4o")
tools = [BashTool(), WriteFileTool(), WebSearchTool()]
agent = create_openai_functions_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)

# No policy enforcement. No audit trail. No kill switch.
# Every tool call goes directly to execution.
# One poisoned web search result = arbitrary code execution.
result = executor.invoke({"input": user_input})

The rest of this guide shows you exactly how to fix this — with a policy layer, monitoring, and a kill switch that doesn't require rebuilding your agent from scratch.

Step 1

Define a Policy

Security starts with a policy: a formal, version-controlled definition of what your agent is allowed to do. Not a system prompt — a policy file. The difference is that system prompts are read by the LLM (and can be overridden by adversarial inputs). Policy files are enforced by a runtime layer that sits outside the LLM.

AgentGuard policies are declared in YAML. Here's a complete policy for a customer service agent with web search and email capabilities:

agentguard-policy.yaml
# AgentGuard Policy — Customer Service Agent v1.2
# Owner: platform-team@company.com
# Last reviewed: 2026-02-28

agent:
  id: "customer-service-agent"
  version: "1.2"
  description: "Handles customer support queries, can search docs and send templated emails"
  owner: "platform-team"

# What this agent is allowed to do
permissions:
  tools:
    - name: "web_search"
      allowed: true
      constraints:
        domains_allowlist: ["docs.company.com", "support.company.com"]
        max_calls_per_session: 10

    - name: "send_email"
      allowed: true
      constraints:
        recipient_domain_allowlist: ["*.company.com"]
        require_template: true
        templates_allowlist: ["support-reply", "ticket-update", "escalation"]
        human_approval_required: false

    - name: "write_file"
      allowed: false  # Explicitly denied

    - name: "bash_tool"
      allowed: false  # Explicitly denied

# Data access controls
data:
  can_access_pii: false
  can_store_conversation: true
  retention_days: 30
  pii_detection: "redact"  # auto-redact PII from logs

# Session limits
limits:
  max_session_duration_minutes: 30
  max_actions_per_session: 50
  max_cost_usd_per_session: 2.00

# High-risk actions requiring human approval
human_in_the_loop:
  require_approval_for:
    - "send_email to external domains"
    - "any action with risk_score > 7"
  approval_timeout_seconds: 300
  timeout_action: "deny"  # fail-closed on timeout

# Anomaly detection thresholds
anomaly_detection:
  alert_on_prompt_injection_score: 0.7
  alert_on_goal_drift_score: 0.6
  auto_halt_on_score: 0.9

This policy file is version-controlled, human-readable, and serves double duty: it's your runtime enforcement config and your Art. 9 risk management documentation for EU AI Act compliance. One source of truth.

💡 Policy-as-Code Philosophy

Treat your agent policy like you treat Terraform or Kubernetes configs — in version control, reviewed via pull requests, deployed through CI/CD. Every change is auditable. This is what "governing AI at scale" actually looks like in practice.

Step 2

Wrap Your Agent

You don't need to rewrite your LangChain agent. AgentGuard wraps your existing AgentExecutor with a security middleware layer. The wrapper intercepts every tool call before execution, evaluates it against your policy, and either allows, blocks, or escalates it.

pip install agentguard-sdk langchain langchain-openai
Python · Secure Agent
import agentguard
from langchain.agents import AgentExecutor, create_openai_functions_agent
from langchain_openai import ChatOpenAI
from langchain.tools import WebSearchTool, SendEmailTool
from langchain.prompts import ChatPromptTemplate

# 1. Initialise AgentGuard with your policy file
guard = agentguard.Guard(
    policy_file="./agentguard-policy.yaml",
    api_key="ag_your_api_key",  # or env var AGENTGUARD_API_KEY
    environment="production"
)

# 2. Build your agent as normal
llm = ChatOpenAI(model="gpt-4o", temperature=0)
raw_tools = [WebSearchTool(), SendEmailTool()]

# 3. Wrap tools with AgentGuard policy enforcement
# Every tool call is intercepted before execution
secured_tools = guard.secure_tools(raw_tools)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a customer service agent for Acme Corp."),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent = create_openai_functions_agent(llm, secured_tools, prompt)

# 4. Create a secured executor — wraps the executor to add chain-of-thought logging
executor = guard.secure_executor(
    AgentExecutor(
        agent=agent,
        tools=secured_tools,
        verbose=False,
        return_intermediate_steps=True  # required for audit trail
    ),
    agent_id="customer-service-agent",
    user_id=user_id,  # for per-user policy and attribution
    session_metadata={"ticket_id": ticket_id, "channel": "web"}
)

# 5. Run exactly as before — security is transparent to your code
try:
    result = executor.invoke({"input": user_message})
    print(result["output"])
except agentguard.PolicyViolationError as e:
    # Policy violation was caught and blocked before execution
    print(f"Action blocked: {e.action} — {e.reason}")
    print(f"Risk score: {e.risk_score}")
except agentguard.KillSwitchError:
    # Agent was halted by kill switch (manual or automatic)
    print("Agent halted. Manual review required.")

What happens under the hood when you call executor.invoke():

  1. AgentGuard intercepts each tool call before LangChain executes it
  2. The tool call is evaluated against your policy file (tool allowlist, parameter constraints, session limits)
  3. Prompt injection detection runs on the inputs to each tool call
  4. Risk scoring runs on the full action context (what's the agent been doing this session?)
  5. If the action passes, it's executed and logged. If it fails, a PolicyViolationError is raised — the tool call never happens
⚡ Latency Impact

The policy evaluation layer adds <50ms p99 overhead. Policy checks are asynchronous for logging; only enforcement is synchronous. Your agent's performance SLA is preserved.

Step 4

Set Up the Kill Switch

The kill switch is the most important control in your runtime security stack — and the most overlooked. It's your ability to halt a running agent mid-session, either manually or automatically, when something goes wrong. EU AI Act Article 14 requires demonstrable human oversight capability. The kill switch is that capability.

AgentGuard provides three kill switch modes:

  • Manual halt — triggered via dashboard, API, or CLI by an operator
  • Automatic halt — triggered when an agent's risk score exceeds a configured threshold
  • Circuit breaker — triggered when a fleet-wide anomaly pattern is detected (e.g., multiple agents behaving abnormally simultaneously)
Python · Kill Switch Configuration
# In your policy YAML (already covers basic auto-halt):
# auto_halt_on_score: 0.9 — configured in Step 1

# Programmatic manual halt — e.g., from your incident response runbook
import agentguard

client = agentguard.Client(api_key="ag_your_api_key")

# Halt a specific session immediately
client.sessions["sess_a7x2"].halt(
    reason="Suspicious exfiltration pattern — manual halt by SOC",
    operator="soc-analyst-01"
)

# Halt ALL sessions for an agent (e.g., security incident)
client.agents["customer-service-agent"].halt_all(
    reason="Incident response — prompt injection campaign detected",
    fail_closed=True  # new sessions also blocked until re-armed
)

# Re-arm after incident review
client.agents["customer-service-agent"].rearm(
    approved_by="security-lead@company.com",
    incident_id="INC-2026-041"
)
Python · Automatic Kill Switch (Event-Driven)
from agentguard import Guard, HaltPolicy

# Configure fine-grained auto-halt rules in code (overrides YAML for dynamic scenarios)
guard = Guard(
    policy_file="./agentguard-policy.yaml",
    halt_policy=HaltPolicy(
        # Halt if risk score exceeds threshold
        risk_score_threshold=0.9,

        # Halt if agent attempts more than N blocked actions in a session
        max_violations_per_session=3,

        # Halt if session exceeds cost limit
        max_cost_usd=5.00,

        # Halt if a specific tool is called with suspicious parameters
        suspicious_tool_patterns=[
            "write_file: path contains /etc",
            "bash_tool: command contains curl",
            "send_email: recipient matches *.external.com",
        ],

        # What to do with the in-flight session when halted
        halt_action="preserve_state",  # save context for forensic review
        notify_webhook="https://your-pagerduty-endpoint/alert"
    )
)
⚠ Test Your Kill Switch

The kill switch you never tested is the kill switch that fails during an incident. Run a quarterly drill: trigger a manual halt in your staging environment, verify the agent stops within the expected time window, and log the test result. Auditors will ask for evidence that oversight controls actually work — not just that they exist.

Putting It All Together

Here's the complete secure LangChain agent pattern — from policy to execution:

Python · Complete Secure Agent Pattern
"""
Secure LangChain Agent — Production Pattern
==========================================
1. Policy from YAML (version-controlled)
2. Tools wrapped by AgentGuard
3. Executor wrapped for audit logging
4. Kill switch configured
5. Compliance reporting enabled
"""
import agentguard
from langchain.agents import AgentExecutor, create_openai_functions_agent
from langchain_openai import ChatOpenAI
from langchain.tools import WebSearchTool, SendEmailTool
from langchain.prompts import ChatPromptTemplate

# Initialise guard from policy file
guard = agentguard.Guard.from_policy("./agentguard-policy.yaml")

# Build agent with secured tools
llm = ChatOpenAI(model="gpt-4o", temperature=0)
tools = guard.secure_tools([WebSearchTool(), SendEmailTool()])
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a customer service agent for Acme Corp."),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])
agent = create_openai_functions_agent(llm, tools, prompt)
executor = guard.secure_executor(
    AgentExecutor(agent=agent, tools=tools, return_intermediate_steps=True),
    agent_id="customer-service-agent"
)

# Run — security, monitoring, and audit trail are automatic
def handle_customer_query(user_message: str, user_id: str, ticket_id: str) -> str:
    with guard.session(user_id=user_id, metadata={"ticket_id": ticket_id}):

            result = executor.invoke({"input": user_message})
