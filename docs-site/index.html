Quick Start

Get from zero to a live policy engine in under 5 minutes.

1. Create an account

curl
curl -X POST https://api.agentguard.tech/api/v1/signup \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Company",
    "email": "security@acme.com",
    "password": "your-secure-password"
  }'

Response includes your apiKey (format: ag_live_ + 32 hex chars). Store it securely — it's only shown once.

Your ag_live_* tenant key works directly with /api/v1/evaluate — you can start evaluating immediately without any additional setup.

3. (Optional) Create scoped agent keys

For production deployments, you can create scoped ag_agent_* keys that are restricted to evaluate only. This is a best practice — it limits the blast radius if a key is leaked.

curl
curl -X POST https://api.agentguard.tech/api/v1/agents \
  -H "Content-Type: application/json" \
  -H "X-API-Key: ag_live_your_key_here" \
  -d '{"name": "booking-agent", "policy_scope": ["booking", "read-only"]}'

The response includes a scoped ag_agent_* key. Agent keys can only call /api/v1/evaluate — they cannot manage webhooks, kill switches, or other tenant resources.

Agent keys are optional. Your ag_live_* tenant key works for all API calls including evaluate. Agent keys are recommended for production multi-agent deployments where you want per-agent audit trails and restricted permissions.

Authentication

All authenticated endpoints require the X-API-Key header.

X-API-Key: ag_live_your_key_here

AgentGuard uses two types of API keys:

Key FormatTypePermissions
ag_live_*tenantFull access — agent management, kill switch, webhooks, policy, all admin operations
ag_agent_*agentRestricted — evaluate, audit, and MCP evaluate only. Policies scoped to this agent.
No API key? Requests without a key run in demo mode — fully functional but data isn't persisted. Great for testing. Demo mode returns a sessionId you can reference for the playground.
Agent keys are restricted. Using an ag_agent_* key on admin endpoints (create agent, kill switch, webhooks) returns 403. Always use your tenant key for management operations.

Local Evaluation Mode

Enable localEval: true to run policy evaluation in-process without a network call. Requires a locally loaded policy file. Useful for offline environments, CI, and minimising latency.

import { AgentGuard } from '@the-bot-club/agentguard';

const guard = new AgentGuard({
  apiKey: 'ag_live_...',
  localEval: true   // evaluate against local policy, no API call for evaluate
});
CJS + ESM supported. The TypeScript SDK works with both require() and import.

Verify Hash Chain

GET /api/v1/audit/verify Auth Required

Cryptographically verify the integrity of your entire audit trail. Re-computes every SHA-256 hash and confirms the chain is unbroken. Returns valid: false if any entry has been tampered with at the database level.

{
  "valid": true,
  "eventsChecked": 42,
  "message": "Hash chain verified — all 42 events are intact."
}
Hash algorithm: SHA-256(previousHash + tool + result + createdAt + tenantId). The first event uses a genesis hash of 64 zeros.

🛡️ Deployment Enforcement

AgentGuard's deployment enforcement layer blocks uncertified agents from reaching production. Integrate into CI/CD pipelines and GitHub Actions to gate every deployment on policy coverage.

The enforcement workflow:

  1. Declare your agent's tools with POST /api/v1/agents/:id/validate
  2. Review coverage — every tool must match a policy rule (100% required)
  3. Certify with POST /api/v1/agents/:id/certify — get a 30-day deployment token
  4. Gate CI/CD on the certification token using the GitHub Action
  5. For MCP servers, run POST /api/v1/mcp/admit before connecting
Why 100% coverage? Any tool without a matching policy rule is a blind spot — AgentGuard can't enforce what it doesn't know about. Certification requires every declared tool to be explicitly covered by at least one policy rule.

Prompt Injection Detection NEW in v0.9.0

AgentGuard's prompt injection detection layer scans incoming messages and conversation histories for adversarial patterns before they reach your agent. Detection combines fast heuristic rules with an optional Lakera Guard adapter for deep semantic analysis.

How It Works

Detection runs in two passes:

  1. Heuristic pass — pattern matching against a curated ruleset covering: instruction overrides ("ignore previous instructions"), role-play jailbreaks, system prompt leakage attempts, indirect injection via tool outputs, and multi-turn escalation sequences.
  2. Semantic pass (optional) — routes to the Lakera Guard API for ML-based classification. Enable by setting LAKERA_API_KEY in your environment.

The messageHistory field lets you pass the full conversation context — enabling detection of multi-turn attacks that look benign in isolation.

Policy Engine

The policy engine evaluates 7 rule types against every action, in priority order. Rules are written in YAML and loaded from templates or custom uploads.

Rule TypeDescriptionExample
resource_accessBlock or allow access to specific tools/resourcesBlock shell_exec, allow file_read
rate_limitLimit action frequency per time windowMax 10 API calls per minute
scope_restrictionRestrict parameter valuesOnly allow http_post to whitelisted domains
hitl_gateRequire human approval before executionFlag any financial action over $1,000
time_windowAllow/block actions based on time of dayNo deployments outside business hours
environmentEnvironment-specific rulesBlock destructive actions in production
data_classificationProtect sensitive data patternsBlock access to PII-classified resources

Rules are evaluated in priority order (lowest number = highest priority). The first matching rule determines the result. If no rule matches, the default action applies (configurable, defaults to allow).

Sub-millisecond evaluation. The policy engine runs entirely in-process — no network calls during evaluation. P99 latency is under 1ms even for complex rule sets.
